# Deploy Smoke Verifier

Drives a **live or preview URL** with Playwright after a deploy to verify that key routes return content, no unhandled console errors appear, and critical user flows still complete. Produces a pass/fail report with per-route status and error details.

**Scope: deployed app correctness.** This skill verifies what is running, not that the CI pipeline succeeded — that is [deployment-watcher](../deployment-watcher/SKILL.md)'s job.

## When to Use

- Immediately after a deployment to staging, preview, or production
- When a CD pipeline succeeds but you want to confirm the app actually behaves
- Before announcing a release as ready for user acceptance testing (UAT)
- After a hotfix deploy to confirm the fix is live and nothing regressed
- When a preview URL is generated by the CI system (Vercel, Netlify, PR previews) and needs quick validation

## Skill Boundaries

- Use this skill to **verify the deployed application at runtime** after a successful deploy.
- If you want to **watch the CI/CD pipeline succeed or fail**, use [deployment-watcher](../deployment-watcher/SKILL.md) first — run that skill before this one.
- If you want **comprehensive E2E user flow tests** authored in advance and run in CI, use [e2e-test-generator](../e2e-test-generator/SKILL.md) to write those tests.
- If you want **performance metrics (LCP, CLS, INP)** from the live URL, use [performance-auditor](../performance-auditor/SKILL.md) after smoke verification passes.
- If you want **SEO meta tag verification** on the live site, use [seo-auditor](../seo-auditor/SKILL.md).

## Prerequisites

- A reachable base URL for the deployed app (environment variable `SMOKE_BASE_URL`, CI output, or user-provided)
- Playwright installed in the environment: `npx playwright install --with-deps chromium`
- If critical flows require authentication: valid test credentials or a pre-generated auth token available as environment variables

## Procedure

1. **Resolve the target URL** — use `SMOKE_BASE_URL` if set; otherwise read the deploy output (Vercel/Netlify preview URL from CI), or ask the user to provide it.
2. **Build the route list** — check for a `smoke-routes.txt` or `smoke-routes.json` in the project root. If absent, derive a default list:
   - `/` (home)
   - `/login` or `/signin` (if detectable from the route tree)
   - Up to five routes inferred from `next.config.*`, `react-router` config, or `src/app/` directory structure
   Ask the user to confirm or extend the list before running.
3. **Navigate and check each route**:
   - Open a new Playwright browser context per route (no shared cookies)
   - Navigate to `<base-url><route>` and wait for `networkidle` or `domcontentloaded`
   - Record HTTP status (via `page.on('response')` for the page document request)
   - Assert the page renders meaningful content: document title is not empty, `<body>` is not blank, no full-page error overlay is visible
   - Collect all `console.error` messages emitted during page load
4. **Run critical path flows** (if user provides or if a `smoke-flows.spec.ts` file exists):
   - Execute each flow in order using the Playwright MCP
   - Record pass/fail per flow with the failing step and screenshot on failure
5. **Compile the report** using the structure in [smoke-report-template.md](./assets/smoke-report-template.md):
   - Overall verdict: PASS / FAIL / PARTIAL
   - Per-route table: route | HTTP status | title present | console errors | verdict
   - Per-flow table (if applicable): flow name | steps completed | failure step | verdict
   - Console error list with route and message text
   - Screenshots for any failed routes or flows

## Output Contract

- **Overall verdict** — PASS (all routes and flows green) / PARTIAL (some warnings) / FAIL (at least one route or flow failed)
- **Route results table** — route | HTTP status | content rendered | console errors count | verdict
- **Flow results table** — flow name | last completed step | failure step | verdict (only if flows were run)
- **Console error log** — grouped by route: error message, source URL, line number
- **Failure screenshots** — one per failed route or flow (base64 or file path)
- **Recommended next steps** — list of errors to investigate, keyed to the relevant skill (e.g. `performance-auditor`, `seo-auditor`)

## Failure Handling

- If the base URL is not reachable (connection refused or DNS failure), stop immediately with a FAIL verdict and surface the URL used so the user can verify the deploy actually completed.
- If Playwright is not installed, emit the install command and pause — do not proceed without a browser.
- If a route returns a 5xx status, mark it FAIL and continue to the next route; do not abort the entire run.
- If authentication is required for a critical flow and credentials are not available, skip that flow, mark it as SKIPPED (not FAIL), and note the missing credentials in the report.
- If more than 50% of routes fail, abort remaining flows and surface the FAIL report immediately — widespread failure indicates a deployment issue, not an individual regression.

## Examples

### Example 1: Vercel preview URL after a PR

User says: "Smoke test the preview at https://myapp-pr42.vercel.app before I merge."

**Actions:**
1. Set base URL to `https://myapp-pr42.vercel.app`.
2. Derive routes: `/`, `/about`, `/login`, `/dashboard` from `src/app/` directory.
3. Navigate to each; confirm 200, non-empty title, zero console errors.
4. Report: all green, PASS verdict.

**Result:** Four-row route table showing PASS on each; overall PASS. PR is safe to merge.

### Example 2: Post-production deploy with auth flow

User says: "We just pushed to prod — verify the login flow still works at https://app.example.com."

**Actions:**
1. Set base URL to `https://app.example.com`.
2. Run route checks on `/`, `/login`, `/dashboard`, `/settings`.
3. Execute the login flow: fill email and password from `SMOKE_EMAIL` / `SMOKE_PASSWORD` env vars, submit, assert redirect to `/dashboard`.
4. `/dashboard` returns 200 but has two `console.error` messages from a third-party analytics script.

**Result:** Route table PASS; flow table PASS; console error log lists the two analytics errors with note "third-party script, not application regression." Overall verdict: PARTIAL with recommendation to investigate analytics initialization.

## Best Practices

- Keep the smoke suite fast (< 60 seconds): check routes and one or two critical flows only. Deep coverage belongs in the E2E suite generated by `e2e-test-generator`.
- Use a dedicated test user account for authenticated flows — never use real user or admin credentials.
- Store the `SMOKE_BASE_URL` and `SMOKE_EMAIL` / `SMOKE_PASSWORD` as CI secrets; never hardcode them.
- Tag `console.error` messages from known third-party scripts as warnings, not failures, to avoid alert fatigue.

## Common Issues and Solutions

### Issue: Page loads but shows a blank white screen

**Cause:** JavaScript bundle failed to parse (syntax error in the deployed build) or a critical API call returned an error that the app does not handle gracefully.
**Solution:** Check the console error log in the report — look for `SyntaxError` or `ChunkLoadError`. Also check the network log for failed API requests. Trigger a rollback if the error is in the app's own bundle.

### Issue: Route returns 200 but Playwright sees a redirect loop

**Cause:** Auth middleware redirects unauthenticated users back to `/login` in a loop due to a cookie misconfiguration in the deployed environment.
**Solution:** Verify that the `SESSION_SECRET` / `AUTH_COOKIE_DOMAIN` environment variables are correctly set for the target environment. Run the login flow with explicit credentials to break the loop.
